## 二、测试点详细说明

### 2.1 TC01 — 登录页渲染

**文件**: `tests/e2e/test_smoke.py::TestLogin::test_login_page_loads`

```
步骤:
  1. 浏览器直访 /admin/login
  2. 断言: token 输入框可见 (data-testid="login-token")
  3. 断言: 登录按钮可见 (data-testid="login-submit")
```

**httpx 测不到的原因**: 无。L1 `test_admin_ui.py` 已有 HTML 断言，E2E 仅加固。

---

### 2.2 TC02 — 正确 token 登录

**文件**: `tests/e2e/test_smoke.py::TestLogin::test_login_with_correct_token_redirects_to_admin`

```
步骤:
  1. 直访 /admin/login
  2. 输入 e2e-token → 点击登录
  3. wait_for_url 到 **/admin
  4. 断言: 会话列表可见 (data-testid="conv-list")
```

**httpx 测不到的原因**: httpx 需要自己维护 cookie/session、模拟跳转。Playwright 验证了真实浏览器内的 302 + cookie 落地 + URL 变化。

---

### 2.3 TC03 — 错误 token 提示

**文件**: `tests/e2e/test_smoke.py::TestLogin::test_login_with_wrong_token_shows_error`

```
步骤:
  1. 直访 /admin/login
  2. 输入 wrong-token → 点击登录
  3. 断言: 错误提示可见 (data-testid="login-error")
  4. 断言: 提示含文本"不正确"
```

**httpx 测不到的原因**: 页面无跳转、无 URL 变化，L1 已测 HTML，E2E 仅确认浏览器可见性。

---

### 2.4 TC04 — 种子会话出现在列表

**文件**: `tests/e2e/test_smoke.py::TestConversation::test_seeded_conversation_appears_in_list`

```
前置: seeded_conversation fixture 在 DB 中写入 Contact + Conversation + Message
步骤:
  1. 登录 → 直访 /admin
  2. 断言: 至少一个会话行可见 (data-testid="conv-row")
  3. 断言: 第一行含文本"E2E User"
```

**httpx 测不到的原因**: 前置数据通过 fixture 写库，E2E 验证从"真实浏览器请求 → SSRF 渲染 → DOM 展开"的完整链路。

---

### 2.5 TC05 — 点会话行 → htmx swap 出气泡（E3）

**文件**: `tests/e2e/test_smoke.py::TestConversation::test_click_conversation_loads_messages`

```
前置: seeded_conversation
步骤:
  1. 登录 → 直访 /admin
  2. 点击第一个会话行 (data-testid="conv-row").first.click()
  3. 等待 htmx 响应: msg-bubble 在 3s 内变为 visible
  4. 断言: 至少一个气泡 (data-testid="msg-bubble")
  5. 断言: 回复输入框可见 (data-testid="reply-input")
  6. 断言: 回复发送按钮可见 (data-testid="reply-submit")
```

**httpx 测不到的原因**: **htmx 局部 swap 是客户端 JS**，httpx 无法触发 `hx-get` 请求，也无法观察 `#message-panel` 内容替换。这是 E2E 的核心价值之一。

---

### 2.6 TC06 — 回复表单 → htmx swap 出 user 气泡（E4）

**文件**: `tests/e2e/test_smoke.py::TestConversation::test_reply_via_htmx_adds_bubble`

```
前置: seeded_conversation
步骤:
  1. 登录 → 直访 /admin
  2. 点会话 → 等 msg-bubble visible
  3. 输入回复内容 (data-testid="reply-input")
  4. 点击发送按钮 (data-testid="reply-submit")
  5. 等待: 3s 内出现 data-sender="user" 的新气泡
  6. 断言: user 气泡数为 1

```

**httpx 测不到的原因**: 同上，htmx POST 驱动 `#message-panel` 替换。L1 只能测到"POST 返回的 HTML 片段正确"，E2E 验证了"片段被真实替换到 DOM 中、新气泡对用户可见"。

### 2.8 TC08 — 登录全程 console 无 error（E8）

**文件**: `tests/e2e/test_smoke.py::TestConsole::test_no_console_errors_during_login_flow`

```
步骤:
  1. 注册 console 事件监听
  2. 登录全流程（直访 → 输入 → 点击 → 等待 admin）
  3. 等待 1s 给 WebSocket onclose 逻辑窗口
  4. 断言: console_errors 为空列表
```

## 四、基础设施说明

### 4.1 测试隔离

```
每个 session（一次 pytest 调用）：
  └─ e2e_server fixture (session-scoped)
       ├─ 创建临时 SQLite 文件
       ├─ create_app(test_config) → uvicorn.run() 在守护线程
       ├─ 等待 /health 就绪
       └─ 返回 base_url

每个测试函数：
  └─ _clean_db fixture (autouse, function-scoped)
       ├─ 清理所有 4 张表（DELETE FROM）
       ├─ 测试执行
       └─ 再次清理所有表
  └─ seeded_conversation fixture (opt-in)
       ├─ 插入 1 条 contact + 1 条 contact_inbox + 1 条 conversation + 1 条 message
       └─ 返回 { conv_id, contact_id }
```

### 4.2 数据库种子数据

Contacts:
- id: `e2e-contact-1`, source_id: `e2e-user-1`, name: "E2E User"

ContactInboxes:
- id: `e2e-ci-1`, inbox_id: `tg`, source_id: `e2e-user-1`

Conversations:
- id: `e2e-conv-1`, inbox_id: `tg`, status: `active`

Messages:
- 1 条 incoming messages: "Hello from E2E test!" (sender: contact)

所有时间戳固定为 `2026-07-07T10:00:00+00:00`，避免时区相关断言问题。

### 4.3 服务器配置

- 端口: 随机空闲（`_find_free_port()`），无冲突
- admin_token: `e2e-token`（与 seed 数据一致）
- channel_type: `test`（不依赖真实 Telegram API）
- DB: 临时文件 SQLite（非内存，TCP 可达）

## 五、data-testid 映射表

所有模板交互元素均标注 `data-testid`，Playwright 通过 `get_by_test_id()` 定位。

| data-testid | 元素 | 所在模板 | 用途 |
|-------------|------|---------|------|
| `login-token` | `<input name="token">` | `login.html` | 登录输入 |
| `login-submit` | `<button type="submit">` | `login.html` | 登录按钮 |
| `login-error` | `<div class="error">` | `login.html` | 登录错误提示 |
| `conv-list` | `<div id="conv-list">` | `admin.html` | 会话列表容器 |
| `conv-row` | `<div class="conv-row">` | `admin.html` | 单个会话行，含 `data-conv-id` |
| `message-panel` | `<div id="message-panel">` | `admin.html` | 右侧消息面板 |
| `conv-status` | `<span id="conv-status">` | `_messages.html` | 会话状态 badge |
| `resolve-btn` | `<button>关闭对话</button>` | `_messages.html` | 关闭对话按钮 |
| `msg-bubble` | `<div class="msg-bubble-*">` | `_messages.html` | 消息气泡，含 `data-sender="contact\|agentbot\|user\|activity"` |
| `reply-input` | `<input name="content">` | `_messages.html` | 回复输入框 |
| `reply-submit` | `<button type="submit">` | `_messages.html` | 发送回复按钮 |
| `reply-error` | `<div class="reply-error">` | `_messages.html` | 回复错误提示 |
| `resolve-error` | `<div class="reply-error">` | `_messages.html` | 关闭操作错误提示 |

## 六、运行方式

### 6.1 本地运行（开发者机器）

```bash
# 首次需安装浏览器
uv run playwright install chromium

# 跑所有 E2E 测试
uv run pytest tests/e2e -v -m e2e

# 跑带失败诊断的完整 E2E
bash script/run_e2e.sh

# 指定单个测试
uv run pytest tests/e2e/test_smoke.py -v -m e2e -k "test_login"
```

### 6.2 CI 自动化

在 CI 中运行 E2E：
```bash
uv run playwright install chromium --with-deps
uv run pytest tests/e2e -v -m e2e
```

### 6.3 失败诊断

Playwright 内置诊断输出：

```
test-results/
└── <test-name>/
    ├── trace.zip        # uv run playwright show-trace trace.zip
    ├── video.webm       # 录屏回放
    └── test-*.png       # 失败瞬态截图
```

命令行使用 `script/run_e2e.sh` 自动生成以上文件。

## 七、注意事项

1. **E2E 不做 HTML 内容断言**: 具体文案留给 L1 测试，E2E 只断言"元素可见、swap 发生、console 干净"。
2. **数据隔离**: 每个测试函数自动清表，`seeded_conversation` fixture 每次都重新写入种子数据。
3. **端口不冲突**: 使用 OS 分配的随机端口（`port=0`），可并行运行。
4. **不依赖外部服务**: 使用 Test adapter（`channel_type="test"`），不调真实 Telegram API。
5. **CI 默认跳过**: `-m "not e2e"` 确保部署钩子不跑浏览器测试。
